# README 更新提醒规则

## 触发时机

在以下场景下，**必须**询问用户是否需要更新 README：

1. 用户执行 `/commit` 指令时
2. 用户明确表示要提交代码时（如「提交代码」「commit」等）
3. 在生成 commit message 并**准备执行提交之前**

## 执行流程

### 1. 检查时机

在 `/commit` 工作流程的**第 5 步（向用户确认）之前**，插入 README 更新检查：

```
步骤 2: 分析代码改动 ✓
步骤 3: 生成 Commit Message ✓
【插入点】→ 步骤 4.5: 检查 README 是否需要更新
步骤 5: 向用户确认 commit message
步骤 6: 执行提交
```

### 2. 询问方式

使用 `ask_user_question` 工具，以友好的方式询问：

```
📋 README 更新检查

本次改动涉及以下内容：
[简要列出核心改动]

是否需要更新 README 或学习进展文档？
```

提供以下选项：

- **需要更新**：暂停 commit 流程，协助更新 README
- **不需要**：继续 commit 流程
- **已更新**：README 已在改动中更新，直接继续

### 3. 根据用户选择执行

#### 选项 1：需要更新

1. 暂停当前 commit 流程
2. 询问用户需要更新哪些内容：
   - 主 README 的学习进展列表
   - 对应章节的 `/docs/xx-xxx/README.md`
   - 其他文档
3. 协助用户完成文档更新
4. 更新完成后，提醒用户重新执行 `/commit`

#### 选项 2：不需要

- 直接继续 commit 流程
- 进入「向用户确认 commit message」步骤

#### 选项 3：已更新

- 确认 README 相关文件已在暂存区
- 直接继续 commit 流程

## 判断标准

### 需要更新 README 的情况

以下改动**强烈建议**更新 README：

1. **新增功能实现**（type: feat）
   - 需要在对应章节的学习进展中记录
   - 更新学到的知识点
2. **遇到并解决了问题**（type: fix）
   - 在「遇到的问题」部分记录
   - 记录解决方案

3. **重要的重构或优化**（type: refactor, perf）
   - 记录重构思路和收获

4. **新增章节**
   - 需要在主 README 的「学习进展记录」中添加新条目
   - 创建对应的章节文档

### 不需要更新 README 的情况

以下改动通常**不需要**更新 README：

1. 代码格式调整（type: style）
2. 测试代码修改（type: test）
3. 构建配置调整（type: build, ci）
4. 琐碎的杂项改动（type: chore）
5. README 本身就在本次改动中

## 特殊说明

### 1. 智能判断

根据改动的 `type` 和 `scope` 智能提示：

- `feat(jsx)` → 提示：「建议在 /docs/01-implement-jsx/README.md 中记录此功能」
- `feat(reconciler)` → 提示：「这是新章节吗？需要创建新的学习进展文档吗？」
- `fix(*)` → 提示：「建议在对应章节的『遇到的问题』部分记录」

### 2. 非侵入性

- 询问应该简洁友好，不打断用户节奏
- 如果用户选择「不需要」，不要重复询问
- 尊重用户的选择，不要强制要求

### 3. 上下文感知

- 如果检测到 README.md 或 docs/ 目录下的文件已在改动中，默认选项设为「已更新」
- 如果是纯文档类改动（type: docs），可以跳过此检查

## 实现要点

1. **时机准确**：必须在生成 commit message 之后、执行 commit 之前
2. **不阻塞流程**：用户选择「不需要」时，立即继续
3. **提供帮助**：用户选择「需要更新」时，主动协助完成更新
4. **记忆用户习惯**：如果用户多次选择「不需要」，可以降低提示频率

## 示例对话

### 场景 1：新增功能

```
[已生成 commit message: feat(reconciler): 实现 Fiber 架构]

📋 README 更新检查

本次改动新增了 Fiber 架构实现（feat(reconciler)）。

建议更新学习进展文档：
- 创建 /docs/02-implement-reconciler/README.md
- 在主 README 添加第二章链接

是否需要更新文档？
[ ] 需要更新（暂停 commit，先更新文档）
[ ] 不需要（直接提交）
[ ] 已更新（文档已在改动中）
```

### 场景 2：修复问题

```
[已生成 commit message: fix(jsx): 修复 children 为 null 时的异常]

📋 README 更新检查

本次修复了一个 bug。建议在 /docs/01-implement-jsx/README.md
的「遇到的问题」部分记录此问题和解决方案。

是否需要更新文档？
```

### 场景 3：已包含文档更新

```
[检测到 README.md 在改动中]

📋 README 更新检查

检测到文档已在本次改动中更新，继续提交流程。
```
